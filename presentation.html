Basic Strategies

The next section will cover some basic strategies to fill in zeros for missing values. These techniques will handle a lot of problems and are pretty easy to implement.

Basic Strategies

  • Show missing items

  • Doesn’t work in this example
If you click on the down arrow on your axis, there will be an option for ‘Show items with no data’. This is always the first option I try. It only works in some situations and doesn’t work in this example.

Basic Strategies

sum_amount_zero = SUM(orders_small[dollar_amount]) + 0
  • Doesn’t fix table
We can just fix the math if you know it’s a constant. All we need to do is add 0 to the end of our measure and it will fill in for us. Unfortunately, this only fixes the graph and not the table.

Basic Strategies

monthly_average = DIVIDE([sum_amount], 12)

  • Doesn’t fix graph
We can fix the table by skipping the DISTINCTCOUNT on months and replace it all with 12. This will get the correct monthly_average. However, this only work when we know it’s always 12. Also, it doesn’t fix the graph.

Basic Strategies

  • Clean up the data by:
  • Importing monthly data
  • Importing zero
Finally, we can always just fix the issues in the data. We can clean up data by importing monthly data from the start and importing rows with zeros from there. There are ways to do that in SQL and Power Query. We won’t go into them in this presentation, but it’s a great option if you know you’ll never need more granular data.

Basic Strategies

  • Tend to work easily
  • But tend to not be extendable
For these strategies, they tend to work very easily. None of them took a lot of effort to implement from where we started. However, they tend to not be extendable. We have to mix and match these strategies to get everything, but that’s annoying. They can fall apart for more advance calculations and become messy when trying to keep it all together. In my experience, if you just need basic graphs and tables, these are great. If you know you’ll more advance calculations and interactivity, a more thorough setup can be better.

Better Strategy

  • Calendar Table Setup
There will be three parts to this one. The first part is setting up a calendar table to use as a date dimension.

Calendar Table

  • Table created in Power BI that contains all the dates you need and associated information, using the CALENDAR function.
  • The CALENDAR function needs a start date and end date.
  • Fills in the day in between (including start and end dates).
  • Three examples
A calendar table is a table created in Power BI that contains all the dates you need and associated information. We’ll use the Calendar Dax function. This function needs start and end dates. It’ll then fill out all the dates in between (including the start and end dates). We’ll go through three examples.

Calendar Table

cal_table = CALENDAR(
  DATE(2020, 01, 01),
  DATE(2020, 12, 31))
cal_table = CALENDAR(
  DATE(YEAR(TODAY()) - 1, 01, 01),
  DATE(YEAR(TODAY()) - 1, 12, 31))
cal_table = CALENDAR(
  MIN(orders_big[date]),
  MAX(orders_big[date]))

We have three different ways to create a date table using the CALENDAR function. The first way provides specific dates. This is useful if you ever need the graph to keep the same date range even if your data changes. The second way uses the TODAY function to get the current date. This will keep your dashboard current no matter what happens with your data. The last way pulls the start and end dates from the data. This will keep your graphs matching the data even if it doesn’t refresh. I tend to use either of the last two depending on the business need.

There are other options like CALENDARAUTO or setting a date table, but for this presentation we’ll stick with CALENDAR. It’s definitely worth your time to spend an afternoon looking into the other options if you’re going to implement this at your work.

The next step for a great setup is using a star schema.

Star Schema

  • Main orders_big table in the middle
  • Important dimensions on the outside
The star schema will work by placing the main orders_big table in the middle. Then have important dimensions on the outside, like our date table. Often you’ll see this has a ‘suggestion’ from Microsoft, but it’s almost always a good idea from the beginning.

Star Schema

product = SUMMARIZECOLUMNS(orders_big[product])
customer = SUMMARIZECOLUMNS(orders_big[customer_id])

Let’s add in two other dimension tables, product and customer. We’ll just summarize the columns from the orders_big table. If they were in other tables in a database, you could bring them in separately. We can sort of see the star shape. It’s just missing the two legs at the bottom. This setup will make our data easy to use with functions.

ADDMISSINGITEMS()

  • Adds rows with empty values to a table returned by SUMMARIZECOLUMNS. (MS Docs)
  • Add the rows with empty measure values back. (dax.guide)
This function is defined as ‘Adds rows with empty values to a table returned by SUMMARIZECOLUMNS.’ from the Microsoft documentation or ‘Add the rows with empty measure values back.’ from the Dax guide. I don’t find either of those definitions intuitive but working through the parameters, output, and examples will help explain what it does.

ADDMISSINGITEMS()

  • ShowAll_ColumnName - What you want to fill in
  • Table - SUMMARIZECOLUMNS table with missing data (based on orders_big table here)
  • GroupBy_ColumnName - What you want to fill in (same as ShowAll_ColumnName unless ROLLUP)
  • FilterTable - Filter out unneeded combinations
There are four parameters. ShowAll_ColumnName is what you want to fill in. The Table parameter will be output from SUMMARIZECOLUMNS with the missing data. It’ll be based on orders_big for us. The GroupBy_ColumnName is also what you want to fill in unless you’re using ROLLUP. We won’t be using ROLLUP in any of the examples, so it’ll always match ShowAll_ColumnName. Finally, you can add filter tables to remove fill in values you don’t actually want to add in.

ADDMISSINGITEMS()

  • ShowAll_ColumnName -
cal_table[cal_month], 'product'[product]
  • Table -
SUMMARIZECOLUMNS(cal_table[cal_month], 'product'[product], 
  "amount", SUM(orders_big[dollar_amount])
For our setup, we’ll want to fill in any missing combinations of month and product. We can pull these from our dimensions in the star schema. The Table parameter is the sum of the dollar_amount by month and product. Ideally, we’ll see all combinations of month and product with the dollar_amount summed, if there were sales that month.

ADDMISSINGITEMS()

orders_zeros_added = 
ADDMISSINGITEMS(cal_table[cal_month], 'product'[product], 
SUMMARIZECOLUMNS(cal_table[cal_month], 'product'[product], 
  "amount", SUM(orders_big[dollar_amount])), 
cal_table[cal_month], 'product'[product])

Putting it all into the ADDMISSINGITEMS function, you can see ShowAll_ColumnName cal_month and product, the Table is the SUMMARIZECOLUMNS, the GroupBy_ColumnName is the same as the ShowAll_ColumnName, no filters for now. This is basically a full join between calendar months and products, then a left join to our summarized data. We can see some of the missing months in the image.

ADDMISSINGITEMS()

orders_zeros_added = 
CALCULATETABLE(ADDCOLUMNS(
ADDMISSINGITEMS(cal_table[cal_month], 'product'[product], 
SUMMARIZECOLUMNS(cal_table[cal_month], 'product'[product], 
  "amount", SUM(orders_big[dollar_amount])), 
cal_table[cal_month], 'product'[product]),
"amount filled", IF(ISBLANK([amount]), 0, [amount])))

Filter Product Availability

  • BLANK is not like SQL NULL
  • Converts to 0 in sums and subtractions
  • Stays BLANK in divisions and multiplication

1 - (A/B) = (B/B) - (A/B) = (B - A)/B

  • If B is BLANK
  • Left side = 1
  • Right side = BLANK
Need to watch this in calculations like the one we just did.
